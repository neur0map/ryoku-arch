## Where logs live

| What | Where |
|---|---|
| Ryoku shell (taskbar, panels, lock-adjacent surfaces) | `journalctl --user -u ryoku-shell` |
| Hyprland (the compositor) | `journalctl --user -u hyprland`, `hyprctl monitors` |
| SDDM (the login screen) | `sudo journalctl -u sddm` |
| Updater | `~/.local/state/quickshell/user/update.log` |
| OpenVPN connections | `journalctl --user -u openvpn@<profile>` |
| Kernel / hardware | `dmesg`, `journalctl -k` |

## The taskbar or panels are not drawing

If core shell surfaces are missing, a shell or Hyprland restart path may have failed.

```bash
journalctl --user -u ryoku-shell -n 50
hyprctl binds
```

The most common causes:

- **Recent `ryoku-update` broke the shell**: `ryoku-snapshot restore` enters the Limine/Snapper restore flow when Snapper is available.
- **Custom config error**: `~/.config/ryoku/shell.json` got malformed (often after hand-editing). The shell will fall back to defaults if you move it aside.
- **Quickshell binary mismatch**: a system update bumped Qt/Quickshell out of sync with the shell sources. Re-running `ryoku-update` usually fixes it.

## Login screen is broken / can't log in

Drop to a TTY (`Ctrl+Alt+F2`), log in, and start a Hyprland session manually:

```bash
Hyprland
```

If Hyprland starts fine from the TTY, SDDM is the problem. Check `sudo journalctl -u sddm` and consider switching SDDM theme (`Settings → Login screen` if you can reach a desktop; otherwise edit `/etc/sddm.conf.d/`).

## Network is offline

- If the shell network toggle does not take effect immediately, check `nmcli device status`.
- If you're tunneled through OpenVPN or Tailscale and the shell says connected but you can't reach anything, `nmcli connection show` and look for a leak or split-routing issue.
- For low-level network state, use `nmcli`, `ip route`, and the relevant VPN service logs.

## Performance feels off

- Animations: `Settings → Appearance → Animations` toggle.
- Wallpaper video / GIF: turning off animated wallpaper is the single biggest GPU win.
- Hyprland animations and blur can be tuned in `~/.config/hypr/hyprland.lua` (or via HyprMod / Settings) if integrated graphics feel choppy.

## Screen sharing is black or has no window to pick

Discord/OBS/Meet share through `xdg-desktop-portal-hyprland`. If the picker never
appears or you only get a black frame:

- Refresh the portal config and restart it:
  `ryoku-refresh-config hypr/xdph.conf` then
  `systemctl --user restart xdg-desktop-portal-hyprland xdg-desktop-portal`.
  (A stale `custom_picker_binary` in `~/.config/hypr/xdph.conf` that points at an
  uninstalled picker breaks source selection.)
- For the official Discord, prefer Vesktop, or log out and back in so
  `ELECTRON_OZONE_PLATFORM_HINT` takes effect (Electron needs Wayland for PipeWire
  capture).
- On a multi-GPU box a black share usually means the compositor and the app are on
  different GPUs; pin one in `Settings → Display → GPU`.

## The wrong GPU is used, or I can't pick a GPU

`Settings → Display → GPU` lists every detected GPU and pins the primary render device
(via `ryoku-gpu`). On a laptop the integrated GPU is the battery-friendly default;
choose the discrete or external GPU there to render the desktop (and streams) on it.
The pin is read at login, so log out and back in to apply it. `ryoku-gpu detect` prints
what was detected and which GPU is recommended.

## Only one resolution, or a display change won't apply

Laptop (eDP) panels expose only their native resolution; use the scale options instead:
they are labelled with the logical resolution they produce (e.g. 2048×1280) and are the
Wayland way to get a smaller desktop. If `Apply` toasts a rejection, the compositor
refused that mode/scale; pick one of the offered (clean-divisor) scales.

## Filing a useful bug

Open an issue at [`neur0map/ryoku-arch`](https://github.com/neur0map/ryoku-arch/issues) with:

1. **What you did** — keystrokes, clicks, or commands.
2. **What happened** — what you saw, including screenshots when relevant.
3. **What you expected**.
4. **Update proof** — include the `Ryoku update result:` block from the last `ryoku-update`, or the output of `ryoku-doctor`.
5. **Relevant log slice** — last 50 lines of `journalctl --user -u ryoku-shell` is usually enough for shell issues.

Security-sensitive issues go through [SECURITY.md](https://github.com/neur0map/ryoku-arch/blob/main/SECURITY.md), not the public tracker.
