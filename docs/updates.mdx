---
title: Updates
description: How Ryoku updates itself, how release channels work, and how to recover a stuck checkout.
---

Ryoku is rolling. The installed checkout at `~/.local/share/ryoku` pulls from
the selected Ryoku channel in [`neur0map/ryoku-arch`](https://github.com/neur0map/ryoku-arch):
`main` for Stable or `unstable-dev` for Unstable. The Arch package set is
rolling too.

## What Gets Updated

- **Installed Ryoku checkout**: `ryoku-update-git` fetches the selected channel
  and updates `~/.local/share/ryoku`.
- **Arch packages**: `pacman -Syu` plus AUR handling through Ryoku's package
  helpers.
- **Ryoku shell**: `install/config/shell.sh` syncs the shell payload into
  `~/.config/quickshell/ryoku-shell`.
- **Migrations**: `ryoku-migrate` runs new scripts under
  `~/.local/share/ryoku/migrations/` once per machine.
- **Doctor and restart steps**: the update flow prints a final proof block and
  offers recovery hints if a step fails.

## Channels

Show the active channel:

```bash
ryoku-channel-current
```

Switch channels:

```bash
ryoku-channel-set main
ryoku-channel-set unstable-dev
```

`ryoku-channel-set` updates the persisted channel, refreshes the matching pacman
mirror configuration, runs the git update for that channel, takes the normal
snapshot when available, and then runs the remaining update stages.

## Updating From A Terminal

Run the normal interactive update:

```bash
ryoku-update
```

Run without the confirmation prompt:

```bash
ryoku-update -y
```

The updater needs sudo authentication for package work. If it is launched from a
non-interactive surface and sudo is not already authorized, run it from a real
terminal or run `sudo -v` first.

Update logs are written to:

```text
~/.local/state/quickshell/user/update.log
```

The final output includes a `Ryoku update result:` block with the installed path,
channel, checkout commit, remote tip, doctor command, active doctor path, and
runtime bridge.

## Rollback

When Snapper is available, Ryoku creates a system snapshot before the package and
migration stages:

```bash
ryoku-snapshot create
```

To enter the configured Limine/Snapper restore flow:

```bash
ryoku-snapshot restore
```

If the shell itself is broken after an update, drop to a TTY (`Ctrl+Alt+F2`),
log in, run the restore command, then reboot or log back into the graphical
session.

## If An Update Fails

The most common failures:

- **No sudo terminal**: run from a real terminal, or run `sudo -v` and retry
  `ryoku-update -y`.
- **Network or remote fetch failure**: check VPN/firewall state and retry.
- **AUR or package conflict**: read the failing package line in the update log,
  then rerun the update after the package conflict is resolved.
- **Migration error**: read `~/.local/state/quickshell/user/update.log` and run
  `ryoku-doctor`.
- **Branch cannot fast-forward**: run `ryoku-doctor update`. The doctor can
  realign old official Ryoku branch history to the selected channel and will
  refuse arbitrary local commits.

## Recovering A Stuck Checkout

If `ryoku-update` says `fatal: Not possible to fast-forward, aborting.`, run:

```bash
ryoku-doctor update
ryoku-update -y
```

For machines that are too stuck to receive the fixed local updater or doctor,
run the current public bootstrap:

```bash
curl -fsSL https://raw.githubusercontent.com/neur0map/ryoku-arch/unstable-dev/bin/ryoku-call911now | bash
```

The bootstrap path does not require a GitHub account. It pulls the installed
Ryoku checkout from the selected channel, repairs stale command shims when it
can, prints the final path/channel/commit proof, and starts the refreshed
updater when the environment is ready.
